# Research notes

Этот документ фиксирует исследовательскую часть проекта для научной статьи.

## Исследовательская цель

Проверить, повышает ли RAG-подход качество подготовки пользователя по загруженным материалам по сравнению с обычным чатом без доступа к источникам.

## Сценарии эксперимента

Сценарий 1. Учебная подготовка:

- пользователь загружает конспект, лекцию, методичку, лабораторную, PDF, DOCX, TXT, MD, PPTX, XLSX или изображение;
- система генерирует краткий пересказ, вопросы, карточки и чат-ответы;
- пользователь проходит тест или задает вопросы.

Сценарий 2. Профсоюзно-конкурсная подготовка:

- пользователь загружает устав, положения, регламенты, федеральные законы, локальные акты;
- система генерирует блиц, 2к1, профсоюзный биатлон, правовое ориентирование;
- пользователь проходит тренировку с таймером и получает разбор ошибок.

## Сравниваемые режимы

Baseline chat:

- модель отвечает без retrieval по документам;
- источники не показываются;
- риск галлюцинаций выше.

RAG chat:

- модель получает только найденные фрагменты документов;
- ответ содержит ссылки на источники;
- при отсутствии основания отвечает: `В загруженных материалах этого нет`.

RAG generation:

- учебные и конкурсные активности создаются только из retrieved chunks;
- каждая активность хранит источники.

## Метрики

Качество ответа:

- accuracy - доля корректных ответов;
- completeness - полнота ответа относительно эталона;
- groundedness - насколько ответ подтверждается источниками;
- hallucination rate - доля утверждений без основания в документах;
- citation precision - доля корректно подобранных источников;
- citation recall - доля нужных источников, которые система нашла.

Пользовательские метрики:

- satisfaction - субъективная удовлетворенность;
- perceived usefulness - насколько ответы помогли подготовке;
- time to answer - время до получения полезного ответа;
- quiz improvement - изменение результата после повторения;
- mistake reduction - снижение повторных ошибок.

Retrieval-метрики:

- top-k relevance;
- mean reciprocal rank;
- chunk hit rate;
- average similarity score;
- no-answer detection rate.

## Что логировать

Для каждого запроса:

- `session_id`;
- сценарий и цель;
- текст запроса;
- collection id;
- document ids;
- source filenames;
- retrieved chunk ids;
- similarity scores;
- prompt template id;
- generated answer/activity id;
- grounding score;
- итог: answered или no_answer.

Для генерации вопросов:

- режим генерации;
- сгенерированные вопросы;
- правильные ответы;
- источники;
- сложность;
- какие чанки использованы.

Для тренировок:

- выбранный режим;
- лимит времени;
- лимит ошибок;
- ответы пользователя;
- ошибки;
- повторные ошибки;
- итоговый score;
- рекомендации.

## Экспериментальный дизайн

1. Подготовить одинаковый набор документов для учебного и конкурсного сценариев.
2. Составить набор контрольных вопросов с эталонными ответами.
3. Прогнать baseline chat и RAG chat по одним и тем же вопросам.
4. Оценить ответы экспертно и автоматически по метрикам.
5. Провести пользовательскую сессию с тестами и анкетой.
6. Сравнить качество, скорость, groundedness и субъективную полезность.

## Гипотезы

- RAG уменьшит hallucination rate по сравнению с обычным чатом.
- Ответы с источниками будут восприниматься как более надежные.
- В конкурсных режимах разбор ошибок улучшит повторный результат.
- Чанкинг с сохранением page/section metadata повысит качество цитирования.

## Риски для валидности

- Качество извлечения текста из PDF, DOCX, PPTX, XLSX и изображений может испортить retrieval.
- Очень маленькие документы могут давать baseline преимущество из-за простоты.
- Очень длинные нормативные документы требуют аккуратного chunking и metadata.
- Экспертная оценка может быть субъективной, поэтому нужна единая рубрика.

## Минимальная рубрика экспертной оценки

Оценивать каждый ответ по шкале 0-2:

- корректность;
- полнота;
- опора на источники;
- отсутствие лишних утверждений;
- ясность объяснения.

Итоговая оценка ответа: сумма баллов / 10.
